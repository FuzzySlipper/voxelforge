# LLM Tool JSON Import Pipeline

Task 754 defines how VoxelForge should import raw LLM tool-call output generated outside the app. The pipeline converts provider/tool JSON into an explicit, replayable VoxelForge artifact and can materialize that artifact into a `.vforge` model for inspection.

This is a design document. It fixes the supported input shapes, preferred import targets, validation behavior, replay/save path, and follow-up implementation split so implementation can proceed without re-opening the format decision.

## Goals

- Accept raw LLM tool-call output from external model runs.
- Normalize supported payloads into a provider-neutral VoxelForge import artifact.
- Replay the normalized artifact through existing headless App/Core/MCP/LLM seams.
- Save a resulting `.vforge` model when the imported operations are valid.
- Preserve enough provenance to inspect what the model emitted and why replay succeeded or failed.
- Report malformed payloads with structured diagnostics instead of partially mutating state.

## Non-goals

- No provider SDK types or provider-specific envelopes in Core.
- No direct mutation of `VoxelModel`, `LabelIndex`, palette, clips, or `.vforge` JSON from raw import data.
- No permissive best-effort model repair in the first version.
- No automatic natural-language interpretation of arbitrary text into tools.
- No renderer, screenshot, FNA, Myra, or Engine dependency.
- No support for every possible MCP or console command in the first version.

## Preferred import target

The first implementation should produce **both** outputs, with one canonical target:

1. Canonical target: a provider-neutral **`VoxelForgeImportPlan`** JSON artifact.
2. Derived target: an optional materialized **`.vforge`** model produced by replaying that plan.

The import plan is the primary artifact because it is inspectable, replayable, diffable, and can preserve raw source locations. A direct `.vforge` file alone loses tool-call provenance and makes validation failures hard to debug. The `.vforge` model is still a required derived output for successful `import`/`materialize` commands because users need to open the result in VoxelForge.

The import plan is not an undoable editor command and is not a Core model format. It is a headless adapter artifact that records ordered external operations and how VoxelForge intends to replay them.

## Supported input shapes

The importer supports four input shape families. Auto-detection is allowed, but explicit `--format` should be available for reproducibility.

### 1. Raw tool envelope

A raw envelope contains a tool name plus arguments. The importer should accept common provider/MCP shapes by normalizing them at the adapter boundary.

Provider-neutral canonical shape:

```json
{
  "name": "set_voxels",
  "arguments": {
    "voxels": [
      { "x": 0, "y": 0, "z": 0, "i": 1 }
    ]
  }
}
```

Accepted aliases in the adapter layer:

```json
{ "tool_name": "set_voxels", "arguments": { "voxels": [] } }
{ "function": { "name": "set_voxels", "arguments": "{\"voxels\":[]}" } }
{ "type": "tool_use", "name": "set_voxels", "input": { "voxels": [] } }
{ "method": "tools/call", "params": { "name": "set_voxels", "arguments": { "voxels": [] } } }
```

The aliases above are parser conveniences only. The normalized import plan stores the provider-neutral name/arguments form.

### 2. Raw tool arguments

Raw arguments are accepted when the caller supplies the tool name explicitly.

Input file:

```json
{
  "voxels": [
    { "x": 0, "y": 0, "z": 0, "i": 1 }
  ]
}
```

CLI:

```bash
dotnet run --project src/VoxelForge.Import -- normalize args.json --tool set_voxels --format raw-arguments --out import-plan.json
```

Raw arguments are useful when a model response contains only a function argument JSON object and the caller already knows which tool was requested.

### 3. Tool-call array or batch

A file may contain an array of envelopes or an object with a `tool_calls` array.

```json
[
  { "name": "set_voxels", "arguments": { "voxels": [{ "x": 0, "y": 0, "z": 0, "i": 1 }] } },
  { "name": "remove_voxels", "arguments": { "positions": [{ "x": 2, "y": 0, "z": 0 }] } }
]
```

```json
{
  "tool_calls": [
    { "id": "call-1", "name": "set_voxels", "arguments": { "voxels": [] } }
  ]
}
```

Batch order is replay order. Import validation must preserve source index and optional call id for diagnostics.

### 4. Command or tool transcript

The importer should accept transcripts generated by the task 753 benchmark design.

`tool-calls.jsonl` line shape:

```json
{
  "index": 1,
  "round": 1,
  "tool_call_id": "call-1",
  "name": "set_voxels",
  "arguments": {
    "voxels": [{ "x": 0, "y": 0, "z": 0, "i": 1 }]
  },
  "ok": true,
  "result_summary": "Set 1 voxel.",
  "duration_ms": 12
}
```

`stdio.jsonl` line shape:

```json
{
  "index": 1,
  "request": { "command": "set", "args": ["0", "0", "0", "1"] },
  "response": { "ok": true, "message": "Set (0,0,0) = 1" }
}
```

For transcript imports, only request/tool-call data drives replay. Prior responses are provenance and may be used to flag mismatches, but the importer must not trust a previous `ok: true` as validation.

## Initial tool and command allowlist

The first implementation should be strict and small. It should normalize and replay only operations needed to materialize model state from common LLM authoring outputs.

### LLM/MCP tools

Supported mutating tools in v1:

- `set_voxels`
- `remove_voxels`
- `apply_voxel_primitives` once the task 752 implementation exists
- `new_model`
- `set_palette_entry`
- `set_grid_hint`
- `create_region`
- `assign_voxels_to_region`
- `delete_region`
- `fill_box`
- `clear_model`

Supported read-only tools may appear in transcripts and should be recorded as skipped no-op operations:

- `describe_model`
- `get_model_info`
- `get_voxel`
- `count_voxels`
- `get_voxels_in_area`
- `list_regions`
- `get_region_voxels`
- `get_region_bounds`
- `get_region_tree`
- spatial query tools such as `measure_distance`, `get_cross_section`, and `check_collision`

Unsupported tools are validation errors by default. A future `--skip-unsupported-readonly` flag can relax this only for known read-only names.

### Stdio console commands

Supported command transcript requests in v1:

- `set`
- `remove`
- `fill`
- `clear`
- `palette`
- `regions`
- `label`
- `grid`
- `undo`
- `redo`

The `save` and `load` console commands should not be replayed from an imported transcript in v1. The import CLI owns input and output paths, and replaying external save/load paths would make imports less deterministic. If a transcript contains `save` or `load`, validation should report an unsupported filesystem command.

## VoxelForgeImportPlan artifact

The normalized plan should use JSON and stable snake_case field names.

```json
{
  "schema_version": 1,
  "source": {
    "format": "tool-calls-jsonl",
    "path": "tool-calls.jsonl",
    "sha256": "...",
    "captured_at_utc": "2026-04-27T00:00:00Z"
  },
  "options": {
    "strict": true,
    "max_operations": 10000,
    "max_generated_voxels": 65536
  },
  "operations": [
    {
      "operation_id": "op-000001",
      "source_index": 1,
      "source_line": 1,
      "source_call_id": "call-1",
      "kind": "tool_call",
      "name": "set_voxels",
      "arguments": {
        "voxels": [{ "x": 0, "y": 0, "z": 0, "i": 1 }]
      },
      "effect": "mutation"
    }
  ]
}
```

`effect` values:

- `mutation`: replay changes model/labels/palette/config.
- `lifecycle`: replay resets or loads the active document in the isolated import session.
- `read_only`: replay skipped by default; kept for provenance.
- `unsupported`: present only in non-strict diagnostic plans; strict mode rejects before writing a successful plan.

The plan preserves the original arguments as JSON because external tool arguments are the interchange artifact. Replay must still route those arguments through existing named tool/command adapters and services; the plan must not become a backdoor model mutation format.

## Replay and materialization path

A successful materialization should use an isolated headless session and write a `.vforge` file through existing project serialization.

Preferred replay path:

```text
Input JSON/transcript
  -> import parser adapter
  -> VoxelForgeImportPlan
  -> import replay runner
  -> existing MCP tool / LLM handler / console command adapter
  -> App services + undoable commands
  -> ProjectLifecycleService / ProjectSerializer
  -> final .vforge
```

For LLM-handler-backed tools such as `set_voxels`, `remove_voxels`, and `apply_voxel_primitives`, replay should use the same handler/application path used by MCP:

```text
IToolHandler.Handle(arguments, model, labels, clips)
  -> ToolHandlerResult.MutationIntent
  -> LlmToolApplicationService.ApplyMutationIntents(...)
  -> VoxelEditingService.ApplyMutationIntent(...)
  -> UndoStack.Execute(CompoundCommand/SetVoxelCommand/...)
```

For typed MCP-only tools such as model lifecycle, palette, and region tools, replay should call the same typed tool/service implementation used by the MCP adapter rather than rebuilding console command strings.

For stdio command transcripts, replay may use `CommandRouter.Execute(commandName, args, context)` because that is the existing stdio front-door shape. Import code should not concatenate command lines.

Replay should validate the full plan before applying any operations. Materialization should write to a temporary output path and atomically move/replace the requested `.vforge` only after replay succeeds.

## CLI shape

Recommended first project:

```text
src/VoxelForge.Import
  VoxelForge.Import.csproj
  Program.cs
  ImportPlanParser.cs
  ImportPlanValidator.cs
  ImportPlanReplayer.cs
  ImportArtifactWriter.cs
```

Commands:

```bash
# Normalize raw JSON/tool transcript into a replayable import plan.
dotnet run --project src/VoxelForge.Import -- normalize raw-tool.json --format auto --out imported.plan.json

# Normalize raw arguments when the tool name is known externally.
dotnet run --project src/VoxelForge.Import -- normalize args.json --format raw-arguments --tool set_voxels --out imported.plan.json

# Replay a plan into a .vforge model.
dotnet run --project src/VoxelForge.Import -- replay imported.plan.json --out content/imported.vforge

# One-shot normalize + replay.
dotnet run --project src/VoxelForge.Import -- import tool-calls.jsonl --format tool-calls-jsonl --plan-out imported.plan.json --out content/imported.vforge

# Validate without writing a model.
dotnet run --project src/VoxelForge.Import -- validate raw-tool.json --format auto
```

Useful flags:

- `--format <auto|tool-envelope|raw-arguments|tool-call-array|tool-calls-jsonl|stdio-jsonl>`
- `--tool <tool-name>` for raw arguments.
- `--strict true|false`, default `true`.
- `--max-operations <n>`, default `10000`.
- `--max-generated-voxels <n>`, default `65536`.
- `--project-dir <dir>` for isolated import materialization.
- `--initial-model <path>` to replay operations on top of an existing `.vforge`.
- `--out <path>` for materialized `.vforge` output.
- `--plan-out <path>` for normalized plan output.
- `--report-out <path>` for validation diagnostics.

## Validation behavior

Validation is two-phase.

### Phase 1: source parsing and normalization

Errors include:

- Invalid JSON.
- JSON Lines file with a malformed line.
- Missing tool name.
- Missing arguments.
- Arguments encoded as a string that is not valid JSON.
- Unsupported input shape when `--format` is explicit.
- Ambiguous input shape when auto-detection cannot choose safely.

Every diagnostic should include:

- severity: `error` or `warning`
- code: stable string such as `IMPORT001`
- message
- source path
- line/column when available
- operation index/call id when available
- JSON pointer when available

### Phase 2: operation validation

Errors include:

- Tool or command not on the allowlist.
- Required fields missing.
- Wrong JSON value type.
- Palette index outside `1..255` where the operation sets material.
- Point coordinate missing or non-integer.
- Generated voxel count above safety cap.
- Region operation references missing region id in the current plan state.
- Filesystem commands such as stdio `save` or `load` in v1.
- Attempted output path outside the requested output directory.

Warnings include:

- Read-only calls skipped during replay.
- Prior transcript response was `ok: false` but the request is still structurally replayable.
- Duplicate voxel assignments where later operations overwrite earlier values.
- `undo`/`redo` operations in command transcripts, because replay depends on exact prior operations.

Strict mode fails on any error and succeeds with warnings. Non-strict mode may produce a diagnostic plan for inspection but must not materialize `.vforge` output if errors remain.

## Error report shape

`import-report.json` should be written for `validate`, failed `normalize`, and failed `import` commands.

```json
{
  "schema_version": 1,
  "status": "failed",
  "source_format": "tool-calls-jsonl",
  "operation_count": 3,
  "accepted_operation_count": 2,
  "skipped_read_only_count": 0,
  "error_count": 1,
  "warning_count": 0,
  "diagnostics": [
    {
      "severity": "error",
      "code": "IMPORT101",
      "message": "Unsupported tool 'save_model' in raw transcript import; use --out to choose the output file.",
      "source_path": "tool-calls.jsonl",
      "line": 7,
      "operation_index": 3,
      "tool_name": "save_model",
      "json_pointer": "/name"
    }
  ]
}
```

Human-readable CLI output should summarize the same diagnostics and point to the JSON report.

## Provider-neutrality rule

Provider-specific envelope support belongs only in the import parser adapter. The normalized plan stores:

- source format label
- tool/command name
- arguments/request data
- optional source ids and source locations

It must not store provider SDK objects or require provider-specific enum values. Core and App services should never branch on provider names.

## Relationship to benchmark artifacts

Task 753 benchmark runs produce `tool-calls.jsonl`, `stdio.jsonl`, final `.vforge`, and metrics. This import pipeline consumes those transcripts when the final `.vforge` is absent, corrupted, or generated outside VoxelForge.

Recommended workflow:

```text
external model run -> tool-calls.jsonl -> VoxelForgeImportPlan -> imported.vforge -> open in VoxelForge / benchmark comparison
```

The import plan can also become a stable fixture for tests and benchmark replay. Comparison tools should prefer `.vforge` plus metrics when available, but can use the plan/transcript to explain model behavior.

## Relationship to existing serialization

The importer should not write `.vforge` JSON by hand. Materialization should end with the same serializer path used by save/load flows:

```text
ProjectLifecycleService.Save or ProjectSerializer.Serialize
```

If the import CLI writes through `ProjectSerializer` directly, it must still obtain state by replaying validated operations through App/Core services. Directly constructing a DTO from raw tool JSON is not allowed.

## Security and path safety

- Input files may be rooted paths because the user is explicitly importing local files.
- Output `.vforge` paths must be explicit and should not be taken from imported payloads.
- Imported `save`, `load`, or model path fields are ignored or rejected in v1.
- `--project-dir` and `--out` path resolution should reject traversal outside the chosen output root unless the user supplies an explicit rooted output path.
- No API keys, provider environment variables, or raw model credentials should be written to import plans or reports.
- Resource caps should protect against huge generated voxel batches and very large transcripts.

## Follow-up implementation split

1. **Import plan schema and parser**
   - Add `src/VoxelForge.Import` console project.
   - Implement format detection plus explicit parsers for raw envelope, raw arguments, tool-call arrays, `tool-calls.jsonl`, and `stdio.jsonl`.
   - Write `VoxelForgeImportPlan` JSON.
   - Add parser/diagnostic tests with malformed fixtures.

2. **Validation and diagnostics**
   - Add allowlist validation, source-location diagnostics, resource caps, and strict/non-strict behavior.
   - Add tests for unsupported tools, malformed arguments, oversized operations, and read-only skips.

3. **Replay/materialization runner**
   - Compose an isolated headless App/Core/MCP session.
   - Replay supported LLM/MCP tools through existing handlers/services.
   - Replay supported stdio commands through `CommandRouter.Execute(commandName, args, context)`.
   - Save final `.vforge` through project serialization.
   - Add tests using small plans and expected voxel counts.

4. **Benchmark transcript integration**
   - Add import tests using task 753-style `tool-calls.jsonl` and `stdio.jsonl` fixtures.
   - Ensure benchmark comparison can link an import plan to materialized `.vforge` output.

5. **Primitive tool import coverage**
   - After task 752 implementation exists, add `apply_voxel_primitives` plan validation and replay tests.
   - Verify primitive imports lower through `VoxelMutationIntent` and undoable commands.

6. **Optional UI/console entry points**
   - Add a thin console command or MCP tool to import a plan into the current session if useful.
   - Keep it as an adapter over the same parser/validator/replay service.

## Acceptance checklist

A follow-up implementation of this design is complete when:

- The supported input shapes are implemented: raw tool envelope, raw arguments with explicit tool name, tool-call batch, task 753 `tool-calls.jsonl`, and task 753 `stdio.jsonl`.
- The canonical output is a `VoxelForgeImportPlan` artifact.
- Successful import can also materialize a `.vforge` through replay and project serialization.
- Validation reports structured diagnostics with source location and operation index.
- Unsupported/malformed input never partially mutates or partially writes a final model.
- Core remains provider-neutral and free of raw JSON/provider envelope dependencies.
- Import code remains headless and does not reference Engine, FNA, Myra, or screenshot APIs.
